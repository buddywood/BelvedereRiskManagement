**Create `src/lib/unified-risk/action-plans.ts`:**

Build a deterministic action plan generator (no AI dependency) that creates prioritized action items from cross-domain risk analysis:

`generateActionPlan(pillarScores: PillarSummary[]): ActionPlanItem[]`:
- Analyze each assessed pillar's score and risk level
- For pillars with HIGH or CRITICAL risk, generate 2-3 action items specific to that domain
- For MEDIUM risk pillars, generate 1 action item
- For LOW risk pillars, generate 0 items (no action needed)
- Use a static recommendation map similar to RISK_RECOMMENDATIONS in intelligence/queries.ts:
  - Governance actions: "Schedule family governance review meeting", "Update succession documentation", "Formalize decision-making framework"
  - Cyber risk actions: "Enable MFA on all financial accounts", "Review password management practices", "Update device security configurations"
  - Identity risk actions: "Audit social media privacy settings", "Review public information exposure", "Implement family digital footprint monitoring"
- Each ActionPlanItem gets a unique id (use `crypto.randomUUID()` or a simple counter-based id like `action-{domain}-{index}`)
- Priority derived from risk level: CRITICAL -> high, HIGH -> high, MEDIUM -> medium
- Status defaults to 'pending'
- Sort final list by priority (high first) then by domain

Also export `ACTION_PLAN_RECOMMENDATIONS` constant for the static recommendation map so it can be tested/extended.

**Create `src/app/api/unified-risk/action-plans/route.ts`:**

POST endpoint -- generates action plan for a specific client:
- Auth: require advisor role (import requireAdvisorRole from advisor/auth)
- Body: `{ clientId: string }`
- Verify advisor-client assignment
- Fetch latest completed assessment with all PillarScore records
- Build PillarSummary array from scores
- Call generateActionPlan(pillarScores)
- Return `{ success: true, data: { actionPlan: ActionPlanItem[], composite: CompositeScore } }`

GET endpoint -- retrieves action plan for a client (same logic, clientId from query param):
- Auth: require advisor role
- Query param: `?clientId=xxx`
- Same logic as POST but idempotent read

Both endpoints follow existing API patterns (try/catch, NextResponse.json, proper status codes 200/400/401/404).
