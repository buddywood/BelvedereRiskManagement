  <name>Task 1: Email notification and API route for intake submission</name>
  <files>
    src/lib/email.ts
    src/app/api/intake/[id]/notify-advisor/route.ts
    src/lib/actions/intake-actions.ts
  </files>
  <action>
1. **Extend src/lib/email.ts** with a new function:
   - `sendAdvisorIntakeNotification(advisorEmail: string, advisorName: string, clientName: string, clientEmail: string, reviewUrl: string)`: Send email via Resend with subject "New Intake Ready for Review - {clientName}". HTML template matching existing Belvedere email style (same bg-#f5f5f5, font stack, button style). Content: "{clientName} ({clientEmail}) has completed their intake interview and is ready for your review." CTA button: "Review Intake" linking to reviewUrl. Same graceful degradation as sendPasswordResetEmail (log error but don't throw, skip if no RESEND_API_KEY).

2. **Create src/app/api/intake/[id]/notify-advisor/route.ts**:
   - POST handler. Authenticate with auth() -- require any authenticated user (the submission flow calls this internally).
   - Params: interviewId from URL (async params pattern for Next.js 15+).
   - Logic:
     a. Look up the IntakeInterview to get userId (the client).
     b. Find ClientAdvisorAssignment(s) where clientId = interview.userId and status = ACTIVE.
     c. For each assigned advisor:
        - Get advisor's User record (email, name) via AdvisorProfile relation.
        - Create in-app AdvisorNotification record with type NEW_INTAKE, referenceId = interviewId.
        - Send email notification via sendAdvisorIntakeNotification.
     d. Return { success: true, notifiedCount }.
   - If no advisor assigned, return { success: true, notifiedCount: 0 } (not an error).
   - Wrap all email sending in try/catch per advisor (one failure doesn't block others).

3. **Update src/lib/actions/intake-actions.ts submitIntakeInterviewAction**:
   - After successful `submitIntakeInterview(interviewId)` call, fire notification.
   - Use fetch to POST to `/api/intake/${interviewId}/notify-advisor` (internal API call).
   - This is fire-and-forget: wrap in try/catch, log errors but never block the submission success response.
   - Do NOT await the notification -- use fetch without await or wrap in a Promise that catches errors silently.
   - The user must NEVER see a submission failure due to notification issues.
  </action>
  <verify>Run `npx tsc --noEmit` passes. Run `npm run build` succeeds. Verify email function exists. Verify API route handles no-advisor case gracefully. Verify submission action still returns success even if notification fails.</verify>
  <done>Intake submission triggers advisor email and in-app notification. Notification failures never block user submission flow.</done>
