  <name>Task 1: Customization pure functions and unit tests</name>
  <files>src/lib/assessment/customization.ts, src/lib/assessment/customization.test.ts</files>
  <action>
Create `src/lib/assessment/customization.ts` with these pure functions:

1. **`getCustomizationConfig(focusAreas: string[])`** — Returns a `CustomizationConfig` object:
   - `isCustomized: boolean` (true when focusAreas has items)
   - `visibleSubCategories: string[]` (the focusAreas array, validated against RISK_AREAS IDs)
   - `emphasisAreas: string[]` (same as visibleSubCategories for now — all visible areas get emphasis)
   - `emphasisMultiplier: 1.5` (constant, per research recommendation)

2. **`getVisibleSubCategories(focusAreas: string[])`** — Filters focusAreas to only valid RISK_AREAS IDs. Returns all 8 subcategory IDs if focusAreas is empty (fallback to standard assessment).

3. **`getEmphasisMultipliers(config: CustomizationConfig)`** — Returns `Record<string, number>` mapping subcategoryId to weight multiplier. Emphasis areas get 1.5x, non-emphasis visible areas get 1.0x. Used by scoring engine.

4. **`estimateCompletionMinutes(visibleSubCategories: string[], allQuestions: Question[])`** — Counts questions in visible subcategories, estimates ~20 seconds per question. Returns estimated minutes. Used for UI display and validation that customization stays within 12-15 minute target. If estimated time exceeds 15 minutes with all categories, return 15 (cap).

5. **`getVisibleQuestionIds(visibleSubCategories: string[], allQuestions: Question[])`** — Returns array of question IDs belonging to the visible subcategories. This is the key filter that controls which questions appear.

Export a `CustomizationConfig` TypeScript interface:
```typescript
export interface CustomizationConfig {
  isCustomized: boolean;
  visibleSubCategories: string[];
  emphasisAreas: string[];
  emphasisMultiplier: number;
  advisorName?: string;
  approvalId?: string;
}
```

Import RISK_AREAS from `@/lib/advisor/types` for validation.
Import Question type from `@/lib/assessment/types` for question filtering.

Create `src/lib/assessment/customization.test.ts` with tests:
- `getCustomizationConfig` with empty focusAreas returns `isCustomized: false` with all 8 subcategories
- `getCustomizationConfig` with 3 valid focusAreas returns `isCustomized: true` with those 3
- `getCustomizationConfig` with invalid IDs filters them out
- `getVisibleQuestionIds` returns only questions matching visible subcategories
- `getEmphasisMultipliers` returns 1.5x for emphasis areas, 1.0x for others
- `estimateCompletionMinutes` returns reasonable range (caps at 15)

Create `src/lib/data/assessment-customization.ts` with:

1. **`getActiveApprovalForUser(userId: string)`** — Queries IntakeApproval where:
   - The interview belongs to the given userId (`interview.userId === userId`)
   - Status is `APPROVED`
   - Returns the most recent approval with: `id`, `focusAreas`, `notes`, `advisor.user.name` (for display)
   - Uses Prisma include to join through IntakeInterview to get user match
   - Returns `null` if no approved intake exists

   ```typescript
   const approval = await prisma.intakeApproval.findFirst({
     where: {
       status: 'APPROVED',
       interview: { userId },
     },
     orderBy: { approvedAt: 'desc' },
     include: {
       advisor: {
         include: {
           user: { select: { name: true } },
         },
       },
     },
   });
   ```

2. **`getCustomizationForUser(userId: string)`** — Calls `getActiveApprovalForUser`, then pipes focusAreas through `getCustomizationConfig` from the customization module. Returns `CustomizationConfig | null`. This is the main entry point for UI components.

- The assessment creation should set `approvalId` if an approval is found, so the scoring engine can reference it later.

Use `'server-only'` import guard on the data file since it accesses Prisma directly.
