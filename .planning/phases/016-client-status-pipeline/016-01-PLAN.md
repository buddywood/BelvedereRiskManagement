**src/lib/pipeline/types.ts** - Define pipeline data types:

```typescript
import { ClientWorkflowStage, InvitationStatus } from '@prisma/client';

// Re-export for convenience
export { ClientWorkflowStage } from '@prisma/client';

// Single client in the pipeline view
export type PipelineClient = {
  id: string;              // User.id
  name: string | null;
  email: string;
  assignedAt: Date;
  stage: ClientWorkflowStage;
  progress: number;        // 0-100 percentage
  lastActivity: Date;      // Most recent status change
  // Invitation data (if exists)
  invitation: {
    status: InvitationStatus;
    sentAt: Date;
    code: string;
  } | null;
  // Intake data (if exists)
  intake: {
    status: string;
    responseCount: number;
    submittedAt: Date | null;
  } | null;
  // Assessment data (if exists)
  assessment: {
    status: string;
    completedAt: Date | null;
    score: number | null;
  } | null;
  // Document tracking
  documents: {
    required: number;
    fulfilled: number;
  };
};

// Summary metrics for the pipeline dashboard
export type PipelineMetrics = {
  total: number;
  byStage: Record<ClientWorkflowStage, number>;
  documentsNeeded: number;  // clients with unfulfilled document requirements
  stalled: number;          // clients with no activity in 7+ days
};

// Filter options for pipeline table
export type PipelineFilters = {
  stage?: ClientWorkflowStage;
  search?: string;
  sortBy?: 'name' | 'stage' | 'progress' | 'lastActivity';
  sortDir?: 'asc' | 'desc';
};
```

**src/lib/pipeline/status.ts** - Stage computation logic:

Create `computeClientStage` function that takes raw client data (invitation, intake, assessment, documents) and returns the appropriate `ClientWorkflowStage`:
- No invitation and no intake → `REGISTERED` (assigned directly)
- Invitation SENT or OPENED → `INVITED`
- Invitation REGISTERED or no invitation + no intake started → `REGISTERED`
- Intake IN_PROGRESS → `INTAKE_IN_PROGRESS`
- Intake COMPLETED or SUBMITTED → `INTAKE_COMPLETE`
- Assessment IN_PROGRESS → `ASSESSMENT_IN_PROGRESS`
- Assessment COMPLETED → `ASSESSMENT_COMPLETE`
- Has unfulfilled document requirements → `DOCUMENTS_REQUIRED`
- All documents fulfilled + assessment complete → `COMPLETE`
- Priority: later stages override earlier ones (assessment > intake > invitation)

Create `computeProgress` function that maps stage to percentage:
- INVITED: 10
- REGISTERED: 20
- INTAKE_IN_PROGRESS: 35
- INTAKE_COMPLETE: 50
- ASSESSMENT_IN_PROGRESS: 65
- ASSESSMENT_COMPLETE: 80
- DOCUMENTS_REQUIRED: 85
- COMPLETE: 100

Create `getStageLabel` function returning human-friendly labels (e.g., "Invited", "Intake In Progress", "Assessment Complete").

Create `getStageOrder` function returning numeric order for sorting (0-7).

Create `isStalled` function: returns true if lastActivity is more than 7 days ago and stage is not COMPLETE.

**src/lib/pipeline/queries.ts** - Pipeline data fetching (mark with `import "server-only"`):

Create `getClientPipeline(advisorProfileId: string): Promise<PipelineClient[]>`:
- Query `clientAdvisorAssignment` where advisorId = advisorProfileId and status = ACTIVE
- Include client user data, clientProfile
- For each assigned client, also query:
  - Latest `InviteCode` where createdBy = advisorProfileId and prefillEmail = client.email (to get invitation status)
  - Latest `IntakeInterview` for the client user
  - Latest completed `Assessment` with scores for the client user
  - `DocumentRequirement` counts where advisorId and clientId match
- Map each to `PipelineClient` using `computeClientStage` and `computeProgress`
- Sort by lastActivity descending by default

Create `getPipelineMetrics(clients: PipelineClient[]): PipelineMetrics`:
- Compute byStage counts from clients array
- Count stalled clients using `isStalled`
- Count documentsNeeded (clients where documents.required > documents.fulfilled)

Follow existing patterns from `src/lib/dashboard/queries.ts` for code style and multi-tenant isolation.
