**src/app/api/auth/register/route.ts:**
Extend the existing POST handler. After the invite token is verified and the user is created:

1. After `consumeInviteCode(inviteCodeId)`, look up the InviteCode to check if it has a `createdBy` (advisor-initiated invitation):
   ```
   const inviteCode = await prisma.inviteCode.findUnique({
     where: { id: inviteCodeId },
     select: { createdBy: true, clientName: true }
   });
   ```

2. If `inviteCode.createdBy` exists (advisor invitation), within a transaction:
   - Create `ClientAdvisorAssignment` linking the new user to the advisor
   - Create `ClientProfile` for the new user (empty, to be filled later)
   - Update the InviteCode status to `REGISTERED` and statusUpdatedAt to now
   - If `inviteCode.clientName` exists, set it as the user's name during creation

3. If `inviteCode.createdBy` is null, the existing non-advisor flow continues unchanged (backward compatible)

4. Add `name` to the register schema as optional: `name: z.string().max(100).optional()`
   - Pass name to user creation if provided

Per locked decision: minimal signup requirements (email, password, name only). The name field enables pre-fill from the invitation.

**src/app/api/invitations/[id]/opened/route.ts:**
Create a POST endpoint that updates invitation status to OPENED:
- Accept invitationId from URL params
- No auth required (client hasn't registered yet)
- Only update if current status is SENT (don't downgrade from REGISTERED)
- Rate limit: simple check that status isn't already OPENED/REGISTERED
- Return 200 OK (don't expose invitation details)
- This will be called by the signup page when it loads with an invitation token
