Accepts: { advisorName, advisorJobTitle, advisorFirmName, advisorEmail, advisorPhone, advisorLicenseNumber, advisorLogoUrl?, personalMessage, invitationUrl, clientName? }

Per locked decisions:
- Personal message format that looks like direct communication from advisor
- Advisor logo only in email header/footer - no complex branding elements
- Security communicated through advisor verification: credentials, firm name, contact info prominently displayed
- Template with customizable fields: greeting, personal note, call-to-action

HTML email structure (inline styles for email client compatibility, following existing src/lib/email.ts patterns):

2. **Greeting:** "Dear {clientName or 'there'}," — personal tone

6. **Divider**

Use `escapeHtml()` helper function (implement in same file) to sanitize advisor-provided personalMessage and clientName. Validate logoUrl with a simple URL pattern check (must start with https://).

Accepts: { clientEmail, advisorInfo (from AdvisorProfile+User), personalMessage, invitationUrl, clientName? }

Follow exact pattern from existing `src/lib/email.ts`:

- Use FROM_EMAIL from env

<task type="auto">
  <name>Task 2: Create validation schema and server actions for invitation lifecycle</name>
  <files>src/lib/schemas/invitation.ts, src/lib/actions/invitations.ts</files>
  <action>
**src/lib/schemas/invitation.ts:**
Create Zod schema `createInvitationSchema`:
- clientEmail: z.string().email("Valid email required")
- clientName: z.string().max(100).optional()
- personalMessage: z.string().max(2000, "Message too long").optional() with a default template like "I'd like to invite you to complete a family governance assessment. This confidential process will help us identify areas where your family's wealth management governance can be strengthened."

**src/lib/actions/invitations.ts:**
Mark with `"use server"`.

1. `sendInvitation(formData: FormData)`:
   - Call `requireAdvisorRole()` to get userId
   - Call `getAdvisorProfileOrThrow(userId)` to get advisor profile with user info
   - Parse and validate form data with createInvitationSchema
   - Call `createAdvisorInvitation(profile.id, validatedInput)` from service (Plan 01)
   - Call `sendAdvisorInvitationEmail(...)` with the invitation URL and advisor info
   - Return `{ success: true, invitation }` or `{ success: false, error: string }`
   - Handle duplicate email gracefully: if client already has pending invitation from this advisor, return helpful error

2. `resendInvitationAction(invitationId: string)`:
   - Call `requireAdvisorRole()` to get userId
   - Get advisor profile
   - Call `resendInvitation(profile.id, invitationId)` from service
   - Re-send the email with new URL using `sendAdvisorInvitationEmail`
   - Return success/error result

3. `expireInvitationAction(invitationId: string)`:
   - Call `requireAdvisorRole()` to get userId
   - Get advisor profile
   - Call `expireInvitation(profile.id, invitationId)` from service
   - Return success/error result

4. `getInvitationsAction(filters?)`:
   - Call `requireAdvisorRole()` to get userId
   - Get advisor profile
   - Call `getAdvisorInvitations(profile.id, filters)` from service
   - Return invitation list

All actions follow existing patterns: auth first, then business logic, then return typed result with success/error.
