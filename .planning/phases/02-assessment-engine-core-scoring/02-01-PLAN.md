  <action>
    **Prisma Schema:** Add to existing schema.prisma (keep all existing models intact):

    ```prisma
    model Assessment {
      id            String           @id @default(cuid())
      userId        String
      version       Int              @default(1)
      status        AssessmentStatus @default(IN_PROGRESS)
      currentPillar String?
      currentQuestionIndex Int?      @default(0)
      startedAt     DateTime         @default(now())
      completedAt   DateTime?
      updatedAt     DateTime         @updatedAt

      user          User             @relation(fields: [userId], references: [id], onDelete: Cascade)
      responses     AssessmentResponse[]
      scores        PillarScore[]

      @@index([userId])
      @@index([status])
    }

    model AssessmentResponse {
      id           String   @id @default(cuid())
      assessmentId String
      questionId   String
      pillar       String
      subCategory  String
      answer       Json
      skipped      Boolean  @default(false)
      answeredAt   DateTime @default(now())
      updatedAt    DateTime @updatedAt

      assessment   Assessment @relation(fields: [assessmentId], references: [id], onDelete: Cascade)

      @@unique([assessmentId, questionId])
      @@index([assessmentId])
    }

    model PillarScore {
      id           String    @id @default(cuid())
      assessmentId String
      pillar       String
      score        Float
      riskLevel    RiskLevel
      breakdown    Json
      missingControls Json?
      calculatedAt DateTime  @default(now())

      assessment   Assessment @relation(fields: [assessmentId], references: [id], onDelete: Cascade)

      @@unique([assessmentId, pillar])
      @@index([assessmentId])
    }

    enum AssessmentStatus {
      IN_PROGRESS
      COMPLETED
      ARCHIVED
    }

    enum RiskLevel {
      LOW
      MEDIUM
      HIGH
      CRITICAL
    }
    ```

    Add `assessments Assessment[]` relation to existing User model.

    Run `npx prisma validate` to confirm schema is valid. Do NOT run `prisma db push` (user handles that).

    **Types (src/lib/assessment/types.ts):** Define comprehensive TypeScript types:
    - `QuestionType`: enum of 'single-choice' | 'yes-no' | 'maturity-scale' | 'numeric' | 'short-text'
    - `Question`: { id, text, helpText?, learnMore?, type, options?, required, pillar, subCategory, weight, scoreMap, branchingRule? }
    - `QuestionOption`: { value, label, description? }
    - `SubCategory`: { id, name, description, weight, questionIds }
    - `Pillar`: { id, name, slug, description, estimatedMinutes, subCategories }
    - `BranchingRule`: { dependsOn, showIf: (answer: unknown) => boolean }
    - `ScoreResult`: { score, riskLevel, breakdown: CategoryScore[], missingControls: MissingControl[] }
    - `CategoryScore`: { categoryId, categoryName, score, weight, maxScore }
    - `MissingControl`: { questionId, category, description, severity: 'high' | 'medium' | 'low', recommendation }
    - `RiskLevel`: 'low' | 'medium' | 'high' | 'critical'

    **Questions (src/lib/assessment/questions.ts):** Define the complete Family Governance question bank. Create 8 sub-categories under the Family Governance pillar:

    1. **Decision-Making Authority** (~8 questions): Formal governance structure, board composition, voting rights, conflict resolution processes, decision documentation
    2. **Access Controls** (~8 questions): Information access policies, financial access controls, digital security, advisor access management
    3. **Trust & Estate Governance** (~10 questions): Trust structure documentation, trustee selection/oversight, beneficiary communication, estate plan currency, review cadence
    4. **Marriage & Relationship Risk** (~8 questions): Prenuptial agreements, asset protection, divorce planning, partner involvement policies
    5. **Succession Planning** (~10 questions): Leadership transition plans, next-gen preparation, role documentation, emergency succession, training programs
    6. **Behavior Standards** (~8 questions): Family constitution/charter, code of conduct, substance/lifestyle policies, social media guidelines, philanthropic governance
    7. **Business Involvement** (~8 questions): Family employment policies, compensation structure, performance reviews, non-family management, ownership transitions
    8. **Documentation & Communication** (~8 questions): Family meetings, record keeping, advisor coordination, dispute resolution procedures, privacy policies

    Each question must have:
    - Unique ID (e.g., 'dma-01', 'ac-02', 'teg-03')
    - Clear plain-language text (per user decision: "conversational, not survey-like")
    - helpText with short inline context sentence
    - learnMore optional expanded explanation
    - Type (mostly single-choice with card-based maturity scales)
    - Options with descriptive labels (e.g., for maturity: "No formal process" / "Informal, case-by-case" / "Documented but not always followed" / "Formal, consistently applied")
    - Weight (1-5, higher = more important to risk score)
    - scoreMap mapping each option value to a 0-10 score (higher = better/lower risk)
    - required flag (most are required; a few per category are optional)

    Export: `familyGovernancePillar` (Pillar object), `allQuestions` (flat Question array), `getQuestionsBySubCategory(subCategoryId)`, `getQuestion(questionId)`.

    IMPORTANT: Unanswered questions must NEVER default to low risk. The scoreMap should not have a default. The scoring engine must handle missing answers separately (per user decision).

    **Scoring (src/lib/assessment/scoring.ts):** Implement hierarchical weighted scoring:
    - `calculatePillarScore(answers: Record<string, unknown>, pillar: Pillar, allQuestions: Question[])`: Returns ScoreResult
    - Algorithm: For each sub-category, calculate weighted average of answered questions only. Then aggregate sub-categories using their weights. Unanswered questions are excluded from average (not treated as 0). Track percentage answered for completeness indicator.
    - `getRiskLevel(score: number)`: Returns RiskLevel. >= 7.5 = low, >= 5.0 = medium, >= 2.5 = high, < 2.5 = critical
    - `identifyMissingControls(answers: Record<string, unknown>, questions: Question[])`: Find questions where answer indicates absence of control (score <= 2 on 0-10 scale). Return sorted by severity (weight * deficit). Top 5 max.

    Score is on 0-10 scale where 10 = excellent governance, 0 = no governance. This inverts the "risk" mental model: LOW risk score means HIGH governance maturity.

    **Branching (src/lib/assessment/branching.ts):** Implement simple 1-level branching:
    - `branchingRules`: Record mapping questionId to BranchingRule
    - `getNextQuestion(currentId, answers, allQuestions)`: Returns next visible question ID or null (end)
    - `getVisibleQuestions(answers, allQuestions)`: Returns filtered array of questions that should be shown given current answers
    - Start simple: questions about trust governance skip if user answered "no trusts". Questions about business involvement skip if user answered "no family business". Questions about succession skip if answered "no children/heirs".
    - Per research: "Start with simple 1-level branching. Monitor completion rates before adding deeper nesting."
