    **Zustand Store (src/lib/assessment/store.ts):**
    Create assessment store using Zustand with persist middleware:
    ```typescript
    interface AssessmentState {
      assessmentId: string | null;
      currentPillar: string | null;
      currentQuestionIndex: number;
      answers: Record<string, unknown>;       // questionId -> answer value
      skippedQuestions: Set<string>;
      completedPillars: string[];
      lastSaved: string | null;
      isLoading: boolean;
      isHydrated: boolean;                    // tracks whether server rehydration is complete

      // Actions
      setAssessmentId: (id: string) => void;
      setAnswer: (questionId: string, answer: unknown) => void;
      skipQuestion: (questionId: string) => void;
      setCurrentPosition: (pillar: string, questionIndex: number) => void;
      markPillarComplete: (pillar: string) => void;
      loadFromServer: (data: ServerAssessmentData) => void;
      resetAssessment: () => void;
      setHydrated: (hydrated: boolean) => void;
    }
    ```

    Use `persist` middleware with `name: 'belvedere-assessment'`. Partialize to only persist: assessmentId, answers, skippedQuestions (serialize as array), completedPillars, currentPillar, currentQuestionIndex, lastSaved. Do NOT persist isLoading, isHydrated, or other transient state.

    **CRITICAL for ASMT-04 (save/resume):** The `loadFromServer` action must accept server assessment data (assessment record + responses array) and overwrite the local store state with server truth. This ensures that even if localStorage is cleared (different browser, cleared cache), the server data is authoritative. The flow is:
    1. On page mount, Zustand rehydrates from localStorage (fast, optimistic)
    2. If assessmentId exists, the hub page fetches from GET /api/assessment/[id] (with responses)
    3. Hub page calls `loadFromServer()` to merge/overwrite with server data
    4. Server data wins for answers and position; this guarantees resume works after browser restart

    Define `ServerAssessmentData` type: `{ id, status, currentPillar, currentQuestionIndex, responses: { questionId, answer, skipped }[] }`.

    Handle SSR safety: Zustand persist middleware handles `typeof window` checks automatically.

    **Providers (src/components/providers.tsx):**
    Create a client component wrapping children with:
    - `QueryClientProvider` from TanStack Query (create QueryClient with sensible defaults: staleTime 5min, retry 1)
    - `Toaster` from react-hot-toast

    Mark as `'use client'`.

    **Update Root Layout (src/app/layout.tsx):**
    Wrap `{children}` with `<Providers>`. Keep all existing layout content intact.

    **Assessment API Routes:**

    `src/app/api/assessment/route.ts`:
    - GET: List assessments for authenticated user. Return assessments with response counts. Requires auth session.
    - POST: Create new assessment for authenticated user. Set status IN_PROGRESS, version 1. Return created assessment with id.
    - Both endpoints check `auth()` session and return 401 if unauthenticated.

    `src/app/api/assessment/[id]/route.ts`:
    - GET: Fetch single assessment by ID with all responses. Verify assessment belongs to authenticated user (return 403 if not). Include responses array with questionId, answer, skipped fields. This endpoint is the source of truth for server rehydration.

    `src/app/api/assessment/[id]/responses/route.ts`:
    - GET: Load all responses for assessment. Verify ownership.
    - POST: Save/upsert a response. Accept `{ questionId, pillar, subCategory, answer, skipped? }`. Use Prisma upsert with `@@unique([assessmentId, questionId])`. Also update assessment's `currentPillar` and `currentQuestionIndex` fields. Return saved response.

    All API routes use `import { auth } from "@/lib/auth"` for session checking and `import { prisma } from "@/lib/db"` for database access. Follow the existing pattern from auth API routes (try/catch, JSON responses, proper status codes).
