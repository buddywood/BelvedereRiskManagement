  <action>
    1. Create `src/lib/encryption.ts`:
       - Use Node.js `crypto` module for AES-256-GCM encryption
       - Export `encrypt(plaintext: string): string` -- generates random IV (16 bytes), encrypts with ENCRYPTION_KEY env var, returns `iv:authTag:ciphertext` as hex string
       - Export `decrypt(encrypted: string): string` -- splits the stored string, decrypts, returns plaintext
       - Key derivation: use `crypto.scryptSync(ENCRYPTION_KEY, 'salt', 32)` to derive a 32-byte key from the env var
       - Throw clear error if ENCRYPTION_KEY env var is missing

    2. Create `src/lib/mfa.ts`:
       - Use `speakeasy` (or `otplib` if speakeasy is unmaintained -- check npm, prefer `otplib` which is actively maintained. Install with `npm install otplib` if needed). Research note suggested `libotp` as fork -- use whichever is available and maintained. If using otplib: `import { authenticator } from "otplib"`.
       - Export `enrollMFA(userId: string)`:
         - Generate TOTP secret using the library
         - Encrypt the secret using `encrypt()` from encryption.ts
         - Store encrypted secret in user.mfaSecret (do NOT enable mfaEnabled yet)
         - Generate QR code URL using `qrcode.toDataURL()` with otpauth URI: `otpauth://totp/Belvedere:${userEmail}?secret=${secret}&issuer=Belvedere`
         - Return `{ qrCodeUrl, secret }` (secret shown as backup for manual entry)
       - Export `verifyMFAToken(userId: string, token: string): Promise<boolean>`:
         - Fetch user.mfaSecret, decrypt it
         - Verify TOTP token with 1 time-step window (allows 30s drift)
         - Return boolean
       - Export `enableMFA(userId: string, token: string)`:
         - Call verifyMFAToken -- if fails, throw error
         - Generate 10 recovery codes: `crypto.randomBytes(4).toString("hex")` each (8-char hex codes)
         - Hash each recovery code with SHA-256 before storing
         - Update user: mfaEnabled=true, mfaRecoveryCodes=JSON array of hashed codes
         - Return the plaintext recovery codes (shown ONCE to user)
       - Export `verifyRecoveryCode(userId: string, code: string): Promise<boolean>`:
         - Hash provided code with SHA-256
         - Check if hash exists in user.mfaRecoveryCodes array
         - If found: remove it from array (single-use), update user, return true
         - If not found: return false

    3. Create `src/app/api/auth/mfa/enroll/route.ts` (POST):
       - Require authenticated session (get session, return 401 if not)
       - Call `enrollMFA(session.user.id)`
       - Return `{ qrCodeUrl, secret }` for display

    4. Create `src/app/api/auth/mfa/verify/route.ts` (POST):
       - Accept `{ token: string, action: "enable" | "login" }` in body
       - If action="enable": call `enableMFA(userId, token)`, return recovery codes
       - If action="login": call `verifyMFAToken(userId, token)`, if valid update session to mark mfaVerified=true. To update session MFA status: either update a custom field in the session table or use a cookie flag. Approach: add `mfaVerified` field to Session model (already in schema from Plan 01), update it via `prisma.session.update()` using the session token from the cookie.
       - Rate limit: 5 attempts per 5 minutes
       - Return success/failure

    5. Create `src/app/api/auth/mfa/recovery/route.ts` (POST):
       - Accept `{ code: string }`
       - Call `verifyRecoveryCode(userId, code)`
       - If valid: mark session as mfaVerified (same as TOTP verify)
       - Return success/failure with remaining recovery code count

    6. Create `src/app/(auth)/mfa/setup/page.tsx`:
       - Client component shown after MFA enrollment initiated
       - Displays QR code image (from enroll API response)
       - Shows secret as text for manual entry
       - Input field for TOTP code verification
       - On submit: POST to /mfa/verify with action="enable"
       - On success: display 10 recovery codes in a copy-friendly format, with strong warning to save them. Show "I've saved my codes" confirmation button.
       - Redirect to /dashboard after confirmation

    7. Create `src/app/(auth)/mfa/verify/page.tsx`:
       - Client component shown after password login when user has mfaEnabled=true
       - Input field for 6-digit TOTP code
       - "Use recovery code" link that toggles to recovery code input
       - On TOTP submit: POST to /mfa/verify with action="login"
       - On recovery submit: POST to /mfa/recovery
       - On success: redirect to originally requested URL (from callbackUrl param or /dashboard)
       - On failure: show error, allow retry

    8. Create `src/app/(protected)/settings/page.tsx`:
       - Server component with session data
       - Sections: Account Info (email, read-only), MFA Status
       - If MFA disabled: "Enable Two-Factor Authentication" button -> navigates to /mfa/setup (triggers enroll API first)
       - If MFA enabled: show "MFA is active" badge, show remaining recovery code count, "Regenerate recovery codes" button (requires TOTP verification)
       - Sign out button

    9. Update `src/middleware.ts` to add MFA enforcement:
       - After checking authentication, check if user has mfaEnabled=true but session.mfaVerified=false
       - If so, redirect to /mfa/verify (except for /mfa/verify itself, /mfa/setup, /settings, and /api/auth/mfa/* routes)
       - This ensures users with MFA enabled must verify TOTP before accessing assessment data
       - To access mfaEnabled and mfaVerified in middleware: the session callback in auth.ts should attach these fields to the session/token. Update auth.ts callbacks if needed to include mfaEnabled from user and mfaVerified from session.

    10. Add ENCRYPTION_KEY to `.env.example` and `.env.local` (generate with `openssl rand -base64 32`).
